> **원칙**: "NestJS면 X 스킬" 같은 프레임워크 매핑이 아니라, "이 프로젝트에서 반복되는 작업 패턴"을 스킬로 만든다.

## 1. 반복 작업 패턴 식별

프로젝트 분석(Phase 2-1)에서 발견한 정보로 "앞으로 반복될 작업"을 예측한다.

### 코드 구조 반복

같은 파일 구성의 디렉토리가 3개+ 반복되면, 새로 만들 일이 있다:
- → `scaffold` 스킬: 프로젝트 컨벤션에 맞는 새 모듈/컴포넌트 골격 생성

### 데이터 스키마 변경 흔적

마이그레이션 디렉토리, 스키마 파일, seed 데이터가 있으면, 스키마 변경이 계속 될 것이다:
- → `migration` 스킬: 스키마 변경 시 마이그레이션 파일 생성·검증

### 테스트 컨벤션 존재

일관된 테스트 파일 패턴이 있으면, 새 코드에도 같은 패턴의 테스트가 필요하다:
- → `test-scaffold` 스킬: 기존 테스트 패턴대로 골격 생성

### CI/CD 파이프라인 존재

파이프라인이 있으면, 코드 변경이 파이프라인에 영향을 줄 수 있다:
- → `pipeline-check` 스킬: 변경 후 파이프라인 영향도 사전 확인

### 환경별 설정 파일 다수

.env.*, config.*.json 등이 여러 개면, 설정 간 불일치가 발생한다:
- → `config-sync` 스킬: 환경별 설정 파일 간 일관성 검증

### 멀티 패키지 구조

모노레포/워크스페이스면, 한 패키지 변경이 다른 패키지에 영향을 준다:
- → `impact-analysis` 스킬: 크로스 패키지 변경 영향도 분석

### API 엔드포인트 다수

API 라우트가 많으면, 변경 시 하위 호환성이 깨질 수 있다:
- → `api-contract` 스킬: API 변경 시 하위 호환성 체크

**위 목록은 예시일 뿐이다.** 탐색 중 발견된 프로젝트 고유의 반복 패턴이 있으면 그에 맞는 스킬을 자유롭게 생성한다.

## 2. 스킬 생성 판단 기준

모든 발견된 패턴이 스킬이 될 필요는 없다.

### 스킬로 만들어야 하는 경우

- 에이전트가 **3회 이상** 동일 패턴의 작업을 할 것으로 예상
- 작업에 **프로젝트 고유의 규칙**이 있어 매번 설명하기 번거로움
- 작업 절차가 **3단계 이상**이고 순서가 중요

### 스킬로 만들지 않아야 하는 경우

- 에이전트가 이미 잘 아는 범용 작업 (Git 커밋, 파일 읽기 등)
- 1회성 작업
- 프로젝트 고유 규칙이 없는 단순 반복

### 적정 스킬 수

| 프로젝트 규모 | 권장 스킬 수 | 이유 |
|-------------|------------|------|
| 소규모 (모듈 5 미만) | 0-2개 | 코어 에이전트의 역량으로 충분 |
| 중규모 (모듈 5-15) | 2-4개 | 반복 패턴이 명확해짐 |
| 대규모 (모듈 15+) | 3-5개 | 반복 패턴 다수, 컨벤션 복잡 |

스킬이 너무 많으면 컨텍스트 윈도우를 압박한다. 각 스킬의 description이 항상 로드되므로, 5개 이하가 적정.

## 3. 스킬 SKILL.md 작성 템플릿

```markdown
---
name: {skill-name}
description: "{이 스킬이 하는 일의 구체적 설명}. {이 스킬을 사용해야 하는 상황 나열}. {트리거 키워드 나열}."
allowed-tools: {필요한 도구 목록}
---

# {Skill Name}

{이 스킬의 목적 한 줄 설명}

## 프로젝트 컨텍스트

{이 스킬이 알아야 하는 프로젝트 고유 정보}
- 소스 루트: `{path}`
- 파일 패턴: `{pattern}`
- 기존 모듈/컴포넌트: {list}

## 워크플로우

### Step 1: {첫 번째 단계}
{구체적 행동 지시}

### Step 2: {두 번째 단계}
{구체적 행동 지시}

### Step 3: {세 번째 단계}
{구체적 행동 지시}

## 출력 형식

{스킬의 산출물 규격}

## 주의사항

{이 프로젝트에서 흔히 발생하는 실수와 회피 방법}
```

### Description 작성 규칙

description은 스킬의 **유일한 트리거 메커니즘**이다. 적극적(pushy)으로 작성한다:

**나쁜 예:**
```yaml
description: "모듈을 생성하는 스킬"
```

**좋은 예:**
```yaml
description: "src/modules/ 하위에 새 모듈 디렉토리를 생성하고 controller/service/repository/dto 구조를 기존 모듈 패턴에 맞춰 스캐폴딩. '모듈 추가', '새 모듈', '모듈 만들어줘', 'add module' 요청 시 반드시 이 스킬 사용."
```

## 4. 프로젝트 컨벤션 적응 방법

스킬 내용은 프로젝트의 **실제 컨벤션**을 반영해야 한다.

### 디렉토리 구조 적응

프로젝트의 기존 모듈을 Read하여 실제 구조를 파악한 후, 스킬의 "프로젝트 컨텍스트" 섹션에 반영한다.

```markdown
## 프로젝트 컨텍스트
기존 user 모듈의 구조를 기준으로 새 모듈을 생성한다:
```
src/modules/user/
├── user.controller.ts    ← @Controller 데코레이터, 라우트 정의
├── user.service.ts       ← @Injectable, 비즈니스 로직
├── user.repository.ts    ← DB 접근 레이어
├── user.module.ts        ← @Module 등록
├── dto/
│   ├── create-user.dto.ts
│   └── update-user.dto.ts
└── user.spec.ts          ← 단위 테스트
```
```

### 네이밍 규칙 적응

코드에서 관찰된 네이밍 패턴을 스킬에 명시:

```markdown
## 네이밍 규칙
- 파일명: kebab-case (`create-order.dto.ts`)
- 클래스: PascalCase (`CreateOrderDto`)
- 변수/함수: camelCase (`findByUserId`)
- 테이블/컬럼: snake_case (`user_id`)
```

### Import 패턴 적응

기존 파일의 import 구조를 분석하여 스킬에 반영:

```markdown
## Import 규칙
- 경로 alias: `@modules/user/user.service` (tsconfig paths 사용)
- 내부 모듈: `import { UserService } from './user.service'`
- 외부 모듈: `import { Injectable } from '@nestjs/common'`
```

## 5. 스킬-에이전트 연결

스킬이 어떤 에이전트에 의해 사용되는지 명시적으로 연결한다.

### 연결 방식

| 방식 | 구현 | 적합한 경우 |
|------|------|-----------|
| **프롬프트 내 인라인** | 에이전트 정의에 스킬 내용 직접 포함 | 스킬이 짧고(50줄 이하) 이 에이전트 전용 |
| **Skill 도구 호출** | 에이전트 프롬프트에 "Skill 도구로 /skill-name 호출" 명시 | 스킬이 독립 워크플로우이고 여러 에이전트가 공유 |
| **레퍼런스 로드** | `Read`로 스킬의 references/ 파일을 필요 시 로드 | 스킬 내용이 크고 조건부로만 필요 |

### CLAUDE.md에 바인딩 기록

하네스 빌드 시 CLAUDE.md에 에이전트-스킬 바인딩을 기록한다:

```markdown
**스킬:**
| 스킬 | 용도 | 사용 에이전트 |
|------|------|-------------|
| scaffold | 새 모듈 스캐폴딩 | executor |
| test-scaffold | 테스트 골격 생성 | test-engineer |
| migration | 마이그레이션 생성·검증 | executor, debugger |
```

## 6. Good vs Bad 스킬 예시

### Good: 프로젝트 맞춤 스킬

```yaml
name: scaffold
description: "src/modules/ 하위에 새 모듈을 생성한다. 기존 user/order/payment 모듈의 패턴(controller/service/repository/module/dto/spec)을 재현. '모듈 추가', '새 모듈', '모듈 만들어줘' 요청 시 반드시 사용."
```

본문에는:
- `src/modules/user/`의 **실제 파일 구조**
- **실제 import 패턴** (`@modules/...`)
- **실제 네이밍 규칙** (kebab-case 파일명, PascalCase 클래스)
- 생성 후 `{name}.module.ts`를 `app.module.ts`에 등록하는 단계

### Bad: 제네릭 스킬

```yaml
name: nestjs-module-scaffold
description: "NestJS 모듈을 생성하는 스킬"
```

본문에는:
- NestJS 공식 문서의 일반적인 모듈 생성 가이드
- `nest g module` 명령어 안내
- 프로젝트의 실제 구조와 무관한 범용 템플릿

**차이점:**
- Good: 이 프로젝트에서만 유효한 구체적 규칙 (경로, 패턴, 네이밍)
- Bad: 어떤 NestJS 프로젝트에나 적용 가능한 범용 가이드 (= Claude가 이미 아는 것)
