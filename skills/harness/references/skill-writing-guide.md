## 1. Description authoring patterns

The description is the only trigger mechanism for a skill. Claude decides whether to use a skill based solely on its name + description in the `available_skills` list.

### Understanding the trigger mechanism

Claude tends not to invoke skills for simple tasks it can handle with its own tools. A simple request like "read this PDF" may not trigger even with a perfect description. The more complex, multi-step, and specialized the task, the more likely the skill triggers.

### Authoring principles

1. State both **what the skill does** and **the concrete trigger situations**
2. Specify boundary conditions that distinguish similar-looking requests that should NOT trigger
3. Be slightly "pushy" — to compensate for Claude's conservative trigger judgment

### Good examples

<!-- user-facing (Korean, do not translate) -->
```yaml
description: "PDF 파일 읽기, 텍스트/테이블 추출, 병합, 분할, 회전, 워터마크,
  암호화/복호화, OCR 등 모든 PDF 작업을 수행. .pdf 파일을 언급하거나
  PDF 산출물을 요청하면 반드시 이 스킬을 사용할 것. 단순히 PDF를
  '읽어달라'는 요청이 아닌 변환/편집/분석이 필요할 때 특히 유용."
```
<!-- /user-facing -->

<!-- user-facing (Korean, do not translate) -->
```yaml
description: "엑셀/CSV/TSV 파일의 열 추가, 수식 계산, 서식, 차트,
  데이터 정제를 포함한 모든 스프레드시트 작업. 사용자가 스프레드시트
  파일을 언급하면 — 심지어 캐주얼하게('다운로드 폴더의 xlsx')라고만
  해도 — 이 스킬을 사용할 것."
```
<!-- /user-facing -->

### Bad examples

- `"데이터를 처리하는 스킬"` — too vague; unclear what file/work
- `"PDF 관련 작업"` — no concrete actions listed, no trigger situations stated

## 2. Body authoring style

### Why-First principle

LLMs make correct edge-case judgments when they understand the why. Conveying context beats coercive rules.

**Bad:**
```markdown
ALWAYS use pdfplumber for table extraction. NEVER use PyPDF2 for tables.
```

**Good:**
<!-- user-facing (Korean, do not translate) -->
```markdown
테이블 추출에는 pdfplumber를 사용한다. PyPDF2는 텍스트 추출에 특화되어
있어 테이블의 행/열 구조를 보존하지 못하기 때문이다. pdfplumber는
셀 경계를 인식하여 구조화된 데이터를 반환한다.
```
<!-- /user-facing -->

### Generalization principle

When feedback or tests reveal a problem, **generalize at the principle level** rather than applying a narrow fix that only matches the specific example.

**Overfit fix:**
<!-- user-facing (Korean, do not translate) -->
```markdown
"Q4 매출" 열이 있으면 해당 열을 숫자로 변환하라.
```
<!-- /user-facing -->

**Generalized fix:**
<!-- user-facing (Korean, do not translate) -->
```markdown
열 이름에 "매출", "금액", "수량" 등 수치를 암시하는 키워드가 있으면
해당 열을 숫자 타입으로 변환한다. 변환 실패 시 원본 값을 유지한다.
```
<!-- /user-facing -->

### Imperative tone

Use "~한다"/"~하라" instead of "~합니다"/"~할 수 있습니다". A skill is a directive.

### Save context

The context window is a shared resource. Ask if every sentence justifies its token cost:
- "Does Claude already know this?" → delete
- "Without this, would Claude make a mistake?" → keep
- "Is one concrete example more effective than a long explanation?" → replace with the example

## 3. Output format definition patterns

Used in skills where the output format matters:

<!-- user-facing (Korean, do not translate) -->
```markdown
## 보고서 구조
다음 템플릿을 정확히 따른다:

# [제목]
## 요약
## 핵심 발견
## 권장 사항
```
<!-- /user-facing -->

Keep the format definition concise — including a real example makes it more effective.

## 4. Example authoring patterns

Examples are more effective than long explanations:

<!-- user-facing (Korean, do not translate) -->
```markdown
## 커밋 메시지 형식

**예시 1:**
입력: JWT 토큰 기반 사용자 인증 추가
출력: feat(auth): JWT 기반 인증 구현

**예시 2:**
입력: 로그인 페이지에서 비밀번호 표시 버튼이 동작하지 않는 버그 수정
출력: fix(login): 비밀번호 표시 토글 버튼 동작 수정
```
<!-- /user-facing -->

## 5. Progressive disclosure patterns

### Pattern 1: domain split

```
bigquery-skill/
├── SKILL.md (overview + domain selection guide)
└── references/
    ├── finance.md (revenue, billing metrics)
    ├── sales.md (opportunities, pipeline)
    └── product.md (API usage, features)
```

When the user asks about revenue, only finance.md is loaded.

### Pattern 2: conditional detail

<!-- user-facing (Korean, do not translate) -->
```markdown
# DOCX 처리

## 문서 생성
docx-js로 새 문서를 생성한다. → [DOCX-JS.md](references/docx-js.md) 참조.

## 문서 편집
단순 편집은 XML을 직접 수정.
**추적 변경이 필요하면**: [REDLINING.md](references/redlining.md) 참조
```
<!-- /user-facing -->

### Pattern 3: structuring large reference files

Reference files of 300+ lines should include a table of contents at the top:

<!-- user-facing (Korean, do not translate) -->
```markdown
# API 레퍼런스

## 목차
1. [인증](#인증)
2. [엔드포인트 목록](#엔드포인트-목록)
3. [에러 코드](#에러-코드)
4. [레이트 리밋](#레이트-리밋)

---

## 인증
...
```
<!-- /user-facing -->

## 6. Script bundling criteria

Watch the agents' transcripts during test runs. The following patterns indicate bundling targets:

| Signal | Action |
|--------|--------|
| Same helper script generated in 3 of 3 tests | Bundle into `scripts/` |
| Same pip install / npm install run every time | State a dependency installation step in the skill |
| Same multi-step approach repeated | Document it as a standard procedure in the skill body |
| Similar errors followed by the same workaround every time | Document known issue and resolution in the skill |

Bundled scripts MUST be tested by execution.

## 7. Data schema standards

Use standard schemas to keep cross-skill data exchange consistent. Usable in tests/evaluations of skills the harness generates.

### eval_metadata.json

Metadata for each test case:

```json
{
  "eval_id": 0,
  "eval_name": "descriptive-name-here",
  "prompt": "the user's task prompt",
  "assertions": [
    "the output contains X",
    "a file was created in Y format"
  ]
}
```

### grading.json

Assertion-based scoring result:

```json
{
  "expectations": [
    {
      "text": "출력에 '서울'이 포함됨",
      "passed": true,
      "evidence": "3번째 단계에서 '서울 지역 데이터 추출' 확인"
    }
  ],
  "summary": {
    "passed": 2,
    "failed": 1,
    "total": 3,
    "pass_rate": 0.67
  }
}
```

**Field name caution:** Use `text`, `passed`, `evidence` exactly (no variants like `name`/`met`/`details`).

### timing.json

Execution time / token measurements:

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

Save `total_tokens` and `duration_ms` immediately from the subagent completion notification. This data is only available at the notification moment and cannot be recovered later.

## 8. What not to include in a skill

- Auxiliary docs like README.md, CHANGELOG.md, INSTALLATION_GUIDE.md
- Meta-information from the skill creation process (test results, iteration history)
- User-facing manuals (a skill is a directive for an AI agent)
- Generic knowledge Claude already has
